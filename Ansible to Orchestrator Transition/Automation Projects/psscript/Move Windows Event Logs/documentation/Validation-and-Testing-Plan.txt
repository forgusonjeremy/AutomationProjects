A1. Confirm PS host is fully configured and registered in vRO
    The Windows PowerShell host must be configured and added to VCF Orchestrator
    before any workflow in this package can be run.

    See: PS-Host-Build-Guide.txt — complete end-to-end configuration guide.

    Key requirements confirmed before proceeding:
      [ ] WinRM configured with HTTPS listener on port 5986
      [ ] HTTPS certificate meets one of the following conditions:
            a) Signed by a CA already trusted by VCF Orchestrator, OR
            b) Self-signed / internal CA certificate imported into the
               VCF Orchestrator trust store BEFORE the PS host was added
               (see PS-Host-Build-Guide.txt Part 2, Step 2.2)
      [ ] PS host added to vRO via "Add a PowerShell host" workflow
          (Library → PowerShell → Configuration) using HTTPS / Kerberos
      [ ] PS host visible in vRO Inventory → PowerShell
      [ ] Smoke test passed: "Invoke a PowerShell script" with Write-Host succeeds

    IMPORTANT — Certificate must be imported before adding the PS host:
    When HTTPS is selected in the "Add a PowerShell host" workflow, vRO
    validates the certificate against its trust store immediately.  The workflow
    does not offer an option to accept or bypass an untrusted certificate.
    If the certificate is not trusted, the workflow will fail with an SSL error.
    Import the certificate first (Step 2.2 in the build guide), then add the host.

A9. Confirm Kerberos constrained delegation (double-hop)
    The PS host reads from \\targetserver\C$\... and writes to \\fileserver\share$\...
    This is a double-hop.  The PS host machine account or service account must be
    trusted for constrained delegation to the file server SPN.
    Check: AD Users and Computers → PS host account → Delegation tab
      [ ] "Trust this computer for delegation to specified services only" is set
      [ ] File server CIFS SPN is listed in the allowed services
    If not configured: engage AD team before running live workflows.
    See also: PS-Host-Build-Guide.txt Step 1.6.

A10. Confirm RSAT AD Tools on PS host
    Run on PS host:
      Get-Module -ListAvailable ActiveDirectory
    Expected: Module is listed.
    If not found: install RSAT AD Tools on PS host before running
    ByADGroupName or ByADGroupCN workflows.
    See also: PS-Host-Build-Guide.txt Step 1.1.

A11. Confirm Remove-OldFiles-UNCShare non-interactive behaviour (WhatIf risk)
    KNOWN ISSUE: Remove-OldFiles-UNCPath calls Read-Host when -Force $false
    (WhatIf='yes').  In a non-interactive PS host session, Read-Host will either
    block indefinitely or return empty string, causing the script to cancel with
    "Operation cancelled by user" and no files listed.

    WhatIf='yes' does NOT function as a safe report-only preview from vRO.

    Before using Remove-OldFiles-UNCShare in any context:
      [ ] Run the script manually on the PS host with WhatIf='yes' to preview scope
      [ ] Only then schedule the vRO workflow with WhatIf='no' for live deletion
      [ ] Document this limitation for operators in the workflow description field

    Phase 2 resolution: add a -ReportOnly switch to Remove-OldFiles-UNCPath that
    lists candidate files without prompting and without deleting.

ROLLBACK AND RECOVERY
───────────────────────────────────────────────────────────────────────────────

Item                      Rollback Approach
────────────────────────  ───────────────────────────────────────────────────────────────────────────
cvs_functions.ps1         No changes required — additive changes were already present in deployed
                          script.  No rollback needed for script.

Moved .evtx files         Files are moved, not deleted.  If a move places files at the wrong
                          destination, they can be manually moved back.
                          The script creates per-server subfolders; match server short name to
                          identify correct destination.

Deleted files (Remove)    No automated recovery — deletion is permanent.
                          Mitigation: always run script manually with WhatIf='yes' first to
                          preview scope before running vRO workflow with WhatIf='no'.

vRO content               All items are new (no modifications to existing vRO content).
                          Rollback = delete the four workflows, five actions, and Config Element.
                          The OOTB "Invoke a PowerShell script" workflow is not modified.

Config Element            Delete or update attribute values to revert defaults.
                          No production impact until workflows are run.

PS host registration      Remove the PS host from vRO Inventory → PowerShell if needed.
                          This does not affect the Windows Server configuration.
                          The Windows Server WinRM configuration can be reverted by:
                            Remove-WSManInstance winrm/config/listener
                              -SelectorSet @{Address="*";Transport="HTTPS"}
                          This removes the HTTPS listener only; HTTP listener is unaffected.
