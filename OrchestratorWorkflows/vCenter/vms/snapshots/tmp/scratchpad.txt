



how do I ssh using a private key?

Use:

ssh -i /path/to/private_key username@hostname
Example:

ssh -i ~/.ssh/id_rsa admin@192.168.1.50
If the key file is too open, SSH will reject it. Fix permissions with:

chmod 600 ~/.ssh/id_rsa
If the server uses a non-default port:

ssh -i ~/.ssh/id_rsa -p 2222 admin@192.168.1.50
If you also have a matching public key already installed on the remote server, that is usually all you need.

On Windows:

In PowerShell with OpenSSH, the same command works.

In PuTTY, you usually need a .ppk key, loaded through PuTTY’s SSH auth settings.

private key is on a windows machine

If you are SSHing from Windows, the command depends on what client you are using.

PowerShell / Command Prompt with OpenSSH
Use the Windows path to the key file:

ssh -i C:\Users\YourUser\.ssh\privatekey username@hostname
Example:

ssh -i C:\Users\Jeremy\.ssh\id_rsa admin@192.168.1.50
If the path has spaces, wrap it in quotes:

ssh -i "C:\Users\Jeremy\My Keys\id_rsa" admin@192.168.1.50
If the server uses another port:

ssh -i "C:\Users\Jeremy\.ssh\id_rsa" -p 2222 admin@192.168.1.50
Git Bash
Use Unix-style path format:

ssh -i /c/Users/Jeremy/.ssh/id_rsa admin@192.168.1.50
PuTTY
PuTTY does not use OpenSSH private keys directly very well unless converted. Usually you need to:

Open PuTTYgen

Load your private key

Save it as a .ppk

In PuTTY, go to:
Connection > SSH > Auth

Browse to the .ppk file

Connect

Important
If your key is in modern OpenSSH format, PowerShell’s built-in ssh usually works best and is less of a headache than PuTTY.

If you are not sure which client you are using, run this in PowerShell:

ssh -V
If it returns an OpenSSH version, use the ssh -i ... format above.


how do I format cloudconfig in VCF Automation 9 VM Apps Org cloud template to create a user, add it to the sudoers group, and provide an ssh public key?

I’m checking the current VCF Automation 9 cloud-init/cloudConfig behavior and the right YAML shape for a VM Apps Org template so the example is accurate.

Thought for 33s
In a VCF Automation 9 VM Apps Org cloud template, put this under the VM resource’s properties.cloudConfig. The content is standard cloud-init YAML, so user creation, sudo privileges, and SSH keys are handled by cloud-init inside the guest. Broadcom’s VCF/Aria Automation cloud templates use cloudConfig for guest initialization, and cloud-init supports users, sudo, and ssh_authorized_keys in that block. 

Example:

formatVersion: 1
inputs:
  vmName:
    type: string
    default: linux-vm
  username:
    type: string
    default: jeremy
  sshPublicKey:
    type: string

resources:
  machine:
    type: Cloud.vSphere.Machine
    properties:
      name: '${input.vmName}'
      image: your-linux-image
      flavor: small
      cloudConfig: |
        #cloud-config
        users:
          - default
          - name: ${input.username}
            shell: /bin/bash
            groups: sudo
            sudo: ["ALL=(ALL) NOPASSWD:ALL"]
            lock_passwd: true
            ssh_authorized_keys:
              - ${input.sshPublicKey}
What matters:

#cloud-config must be the first line.

users: defines the local account.

sudo: is what actually gives sudo rights. Cloud-init accepts strings or lists like ALL=(ALL) NOPASSWD:ALL. 

ssh_authorized_keys: is the correct cloud-init field for the public key. 

lock_passwd: true is common when you only want key-based SSH and no password login.

One important correction: “sudoers group” is usually not the right Linux group name.

On Ubuntu, use groups: sudo

On RHEL/Rocky/Alma/CentOS, use groups: wheel

For RHEL-style images, use this instead:

cloudConfig: |
  #cloud-config
  users:
    - default
    - name: ${input.username}
      shell: /bin/bash
      groups: wheel
      sudo: ["ALL=(ALL) NOPASSWD:ALL"]
      lock_passwd: true
      ssh_authorized_keys:
        - ${input.sshPublicKey}
Red Hat’s cloud-init docs explicitly show groups: wheel and sudo: ["ALL=(ALL) NOPASSWD:ALL"] for sudo-capable users. 

Two things that commonly break this in VCF Automation:

Wrong indentation inside the multiline cloudConfig block.

Image mismatch: the guest image must actually support cloud-init, or none of this will run. Broadcom notes that image-map initialization and template cloudConfig can merge, and conflicting entries can cause unexpected behavior.

Yes, provided SSH itself is enabled on the VM and the public key you supplied matches the private key you hold.

What that cloudConfig does is:

create the user

allow that user to authenticate with the matching private key

grant sudo rights after login

What it does not guarantee by itself:

that the SSH server is installed

that the SSH service is running

that password login is enabled

that firewalls/security groups allow TCP 22

that the image isn’t blocking SSH for that user

So in practice:

On most cloud-init-ready Linux images, yes, that user will be able to SSH in with the private key.

But only if the template/image already has OpenSSH server installed and running.

You would log in like this:

ssh -i /path/to/private_key username@vm-ip
If you want to make it more explicit, you can add package/service setup in cloudConfig.

For Ubuntu/Debian-type images:

cloudConfig: |
  #cloud-config
  packages:
    - openssh-server

  users:
    - default
    - name: ${input.username}
      shell: /bin/bash
      groups: sudo
      sudo: ["ALL=(ALL) NOPASSWD:ALL"]
      lock_passwd: true
      ssh_authorized_keys:
        - ${input.sshPublicKey}

  runcmd:
    - systemctl enable ssh
    - systemctl restart ssh
For RHEL/Rocky/Alma/CentOS:

cloudConfig: |
  #cloud-config
  packages:
    - openssh-server

  users:
    - default
    - name: ${input.username}
      shell: /bin/bash
      groups: wheel
      sudo: ["ALL=(ALL) NOPASSWD:ALL"]
      lock_passwd: true
      ssh_authorized_keys:
        - ${input.sshPublicKey}

  runcmd:
    - systemctl enable sshd
    - systemctl restart sshd
One important point: if you set lock_passwd: true, that user is generally meant for key-based login only, not password SSH login.

The blunt answer is: it authorizes SSH key login for that user, but it does not by itself guarantee that SSH access is available end to end.

Broadcom’s docs say that when you use cloud-init network assignment commands in your cloud template, you must set customizeGuestOs: false so vSphere guest customization does not run. Broadcom also documents cloudConfig / cloud-init initialization for vSphere templates as a separate mechanism from traditional guest customization.

What’s happening in your case is:

your VM is successfully consuming OVF seed ISO data via cloud-init (DataSourceOVF [seed=iso]) and finishing in about a minute. 


but vCenter customization is still being invoked, which is why the vCenter Events pane shows customization started and why VCF Automation stays in progress waiting for that path to finish.

So the solution is not inside CentOS anymore. It is in the cloud template design.

What to change
If you want cloud-init to own initialization, then move the network settings into cloudConfig and set:

customizeGuestOs: false
If you leave networking in the networks: section with assignment: static, VCF Automation/vSphere will continue using guest OS customization for network/guest settings, and vCenter will continue to think it is customizing the guest. That is consistent with what you’re seeing in Events.

Why this matters
Right now you have a mixed model:

cloud-init via OVF seed ISO is doing user creation and hostname

vCenter guest customization is still being triggered for the VM

That mixed model is exactly why:

the guest is actually done quickly

but the deployment stays open waiting on vCenter customization status

The clean fix
Use one model only.

For your goal, that means:

formatVersion: 1
inputs:
  vmName:
    type: string
  vmFolder:
    type: string
  datastoreName:
    type: string
    enum:
      - cvsdev-cl01-ds-vsan01
      - ds1
      - ds2
      - ds3
  sshKey:
    type: string
  ipAddress:
    type: string
  prefixLength:
    type: integer
    default: 24
  gateway:
    type: string
  dnsServers:
    type: array

resources:
  Cloud_vSphere_Machine_1:
    type: Cloud.vSphere.Machine
    properties:
      name: ${input.vmName}
      folderName: ${input.vmFolder}
      image: CentOS
      cpuCount: 2
      totalMemoryMB: 2048
      customizeGuestOs: false
      storage:
        constraints:
          - tag: dsName:${input.datastoreName}
      networks:
        - network: ${resource.Cloud_vSphere_Network_1.id}
      cloudConfig: |
        #cloud-config
        preserve_hostname: false
        hostname: ${input.vmName}
        create_hostname_file: true
        manage_etc_hosts: localhost

        users:
          - name: sshuser
            shell: /bin/bash
            groups: wheel
            sudo: ["ALL=(ALL) NOPASSWD:ALL"]
            lock_passwd: true
            ssh_authorized_keys:
              - "${input.sshKey}"

        write_files:
          - path: /etc/NetworkManager/system-connections/ens33.nmconnection
            permissions: '0600'
            owner: root:root
            content: |
              [connection]
              id=ens33
              type=ethernet
              interface-name=ens33
              autoconnect=true

              [ipv4]
              method=manual
              address1=${input.ipAddress}/${input.prefixLength},${input.gateway}
              dns=${join(input.dnsServers, ";")}
              may-fail=false

              [ipv6]
              method=ignore

        runcmd:
          - nmcli connection reload
          - nmcli connection up ens33 || nmcli connection up filename /etc/NetworkManager/system-connections/ens33.nmconnection
That is the pattern that matches Broadcom’s guidance: if cloud-init is doing the guest/network init, disable guest OS customization.
