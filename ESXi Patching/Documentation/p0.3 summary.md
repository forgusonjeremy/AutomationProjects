## 2. P0.2 — Configuration Element Schemas (Final)

All CEs under category path: **`com.broadcom.pso.vc.esxi.remediation`**

### CE Inventory (5 total)

| CE | Purpose | Lifecycle | Concurrency Risk |
|---|---|---|---|
| `ClusterLocks` | Active cluster lock tracking | Transient (per cluster run) | Medium |
| `PatchCompatibilityMatrix` | Memoized bundle metadata | Persistent cache | Low |
| `RemediationDefaults` | Tunable parameters | Static configuration | None |
| `RemediationHistory` | Active state tracker (in-flight + retained failures) | Transient + retained failures | **High** |
| `HostRemediationLedger` | Persistent per-host last-attempt record | Persistent (per-host upsert) | Medium |

### Key Lifecycle Rules

**`RemediationHistory`**:
- Insert at per-host workflow `H0_INIT`
- If existing entry for `hostMoRef` is `Failed` or `Unrecoverable` → delete and recreate (replacement-on-retry rule)
- Update at each state transition
- Remove at parent `S7_REPORT` after report dispatch confirmed for `Success` / `SuccessWithWarnings`
- Remove at completion for `Skipped` (data captured in report before purge)
- **Retain** entries with status `Failed` or `Unrecoverable` until manual clear, Cleanup Workflow action, or successful retry
- Parent workflow attribute `historyConfigElementName` holds literal string `"RemediationHistory"` (no runtime CE discovery)
- `consecutiveFailureCount` removed (replaced-on-retry rule made it meaningless); `retryAttemptInRun` per-run counter for `repeatFailureThreshold` enforcement

**`HostRemediationLedger`**:
- Inserted/upserted by parent at `S7_REPORT` after each host's terminal status determined
- Never purged by remediation workflow
- Read by Cleanup Workflow to determine whether observed host state (SSH enabled, maintenance mode, ephemeral account) resulted from remediation or admin action
- Captures `lastAttemptCompletedAt`, `lastAttemptEnabledSSH`, `lastAttemptEnteredMaintenance`, `lastAttemptCreatedEphemeralAccount` for drift correlation

### `RemediationDefaults` Tunables (Generic Placeholders)

Categories: schema metadata, notification (shared distribution with separate subject prefixes for remediation vs cleanup), SSH/ephemeral account, host operation timeouts, workflow behavior, Cleanup Workflow.

Notable defaults:
- `repeatFailureThreshold = 2` (per-run retry cap; per-run scope only)
- `serviceExemptions = []` (placeholder; expected element shape documented)
- `dryRunMode = false` (defensive global kill-switch; recommend `true` for P0/P1 lab)
- `parallelClusterLimit = 4`
- `cleanupReportOnNoFindings = false` (suppress empty cleanup reports)
- `cleanupNotificationSubjectPrefix = "[ESXi Cleanup]"` vs `remediationNotificationSubjectPrefix = "[ESXi Remediation]"`

### `util.log` Design

| Property | Value |
|---|---|
| Module | `com.broadcom.pso.vc.esxi.remediation.util` |
| Inputs | `level: string`, `message: string`, `context: Properties` (optional) |
| Returns | `void` |
| Behavior | Validates level, serializes context to `key=value` pairs, dispatches to appropriate `System.*` method |
| Marker | NOT set inside `util.log` — handled separately |

### Marker Strategy (`System.setLogMarker`)

- Set once in **entry-point workflows only** as the first scriptable task
- Children inherit; do not re-set
- Format: `WorkflowName:<workflowName>-WorkflowRunId:<workflow.id>`
- Entry-point workflows: parent (`#Install Offline-Depot.zip in Clusters in vCenter`), `#Cleanup ESXi Remediation Drift on Hosts`, all Operational workflows
- Per-cluster and per-host workflows do NOT set marker (inherit from parent)
- Per-call lineage IDs (`parentRunId`, `clusterRunId`, `hostRunId`, `hostFqdn`) flow via `util.log` `context` Properties bag

### Cleanup Workflow Scope (Expanded from "Orphan Sweeper")

Drift Reconciliation with administrator-action awareness:

| Check | Condition | Action | Report Treatment |
|---|---|---|---|
| Ephemeral account `vrorem-*` exists | Always remove if found | Delete account (cascades permissions) | Log as cleanup |
| SSH enabled, started before last remediation `lastAttemptCompletedAt` | Service was started by remediation | Disable SSH | Log as cleanup |
| SSH enabled, started after `lastAttemptCompletedAt` | Admin-initiated post-remediation | Do NOT disable | Annotate, flag in report |
| Host in maint mode, entered during last remediation window | Likely stuck from failed remediation | Subject to recovery logic | Log as orphan |
| Host in maint mode, entered after `lastAttemptCompletedAt` | Admin-initiated | Do NOT exit | Annotate, flag in report |

Correlation against last remediation attempt uses `HostRemediationLedger` (survives `RemediationHistory` purge).

## 5. Architecture Snapshot (Unchanged from Prior)

### Platform

- VCF Automation 9 with embedded Orchestrator
- Target: ESXi 8.0 U3, vSphere 8 vCenter
- Cluster types: vSAN (VxRail-managed) + VMFS (PowerFlex)
- SDDC Manager out of scope; VxRail Manager bypassed
- Patch source: Content Library offline bundle ZIP per vCenter
- Install: esxcli via temp local account over SSH
- Shared vRO vCenter endpoint with admin privileges (no project-specific account/role)

### Workflow Hierarchy

Service Broker form (action bindings) → Parent (`#Install Offline-Depot.zip in Clusters in vCenter`) → Per-Cluster (`#Install Offline-Depot.zip in Cluster in vCenter`, parallel) → Per-Host (`#Install Offline-Depot.zip on Host in Cluster`, serial)

### State Machine Tiers

- Parent: `S0_INIT` → `S7_REPORT`
- Per-Cluster: `C0_INIT` → `C9_DONE`
- Per-Host: `H0_INIT` → `H13_CLEANUP`, with rollback states `R1`–`R5`

### Status Taxonomy

`Success` | `SuccessWithWarnings` | `Skipped` | `Failed` | `Unrecoverable`

### Pre-Flight Gate Responsibility

Service Catalog custom form must prevent cluster selection if `ClusterLocks` entry exists for that cluster. Lock CE is authoritative concurrency gate, checked at form load time and re-validated at workflow start (TOCTOU defense). vRO action `listAvailableClusters` filters against current `ClusterLocks` for form data binding.
