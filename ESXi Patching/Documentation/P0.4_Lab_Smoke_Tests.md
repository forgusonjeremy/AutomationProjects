## Test 1 — Mail Plugin Smoke Test

**Mutation**: No (sends one email; no vSphere or lab infrastructure state changes)
**Validates**: vRO Mail plugin can dispatch via the configured SMTP relay; output reaches the recipient mailbox.

### Prerequisites

- SMTP relay configured in vRO Control Center (external dependency)
- Test recipient mailbox accessible to operator
- vRO Client open with Workflow editing permissions

### Procedure

1. In vRO Client, create a temporary workflow named `tmp-P0.4-Test1-MailSmoke` in a scratch folder.
2. Add a single Scriptable Task with the following script. Replace placeholders with values from Pre-Execution Environment Capture.

   ```javascript
   // P0.4 Test 1 — Mail Plugin Smoke Test
   var mail = new EmailMessage();
   mail.smtpHost = "<smtp-relay-fqdn>";
   mail.smtpPort = 25; // adjust if relay requires different port

   mail.toAddress = "<test-recipient>";
   ```

### Expected Result

## Test 2 — vRO → ESXi TCP/22 Reachability

**Mutation**: No (read-only network probe)
**Validates**: vRO appliance can establish a TCP/22 connection to the designated lab ESXi host. This is the network path the SSH plugin will traverse in production.

### Prerequisites

- SSH access from vRO management network to ESXi management network permitted (firewall, NSX DFW, etc.)
- Designated lab ESXi host FQDN known
- Note: ESXi SSH service does **not** need to be running for this test — TCP-layer reachability is checked, not SSH protocol negotiation.

### Procedure

1. SSH to the vRO appliance as `root` (or use the embedded vRO appliance shell access mechanism in VCF Automation 9).
2. Execute:

   ```bash
   nc -zv <esxi-fqdn> 22
   ```

3. Capture the full command output.
4. If `nc` reports `Connection refused`, ESXi is reachable but SSH service is stopped — this is the expected default state and **counts as PASS** for this test (TCP layer reachable).
5. If `nc` reports `Connection timed out` or `No route to host`, the network path is blocked.

### Expected Result

- `nc` output indicates either `succeeded` / `open` (SSH service running) or `Connection refused` (SSH service stopped, but host reachable).

### Pass/Fail Criteria

- **PASS**: Output indicates either successful connection or explicit refusal at the host.
- **FAIL**: Output indicates timeout, no route, or unreachable destination.

### On Failure (BLOCK)

Test 6 (SSH plugin smoke) and Test 7 (which uses SSH-related plugin methods) cannot proceed. Engage network team to validate firewall path between vRO appliance and ESXi management network.

### Rollback

None required.

### Evidence

- Text capture of `nc` command and output

## Test 5 — ESXi Service Start Timestamp via vSphere API

**Mutation**: No (read-only property inspection)
**Validates**: Whether the vSphere API exposes the start timestamp of an ESXi host service (specifically `TSM-SSH`) as a queryable property. This determines the Cleanup Workflow's correlation logic for "SSH started before/after last remediation."

### Prerequisites

- Designated lab ESXi host FQDN known
- vCenter endpoint registered in vRO inventory
- Note: SSH service may be stopped or running — this test inspects the property regardless.

### Procedure

1. In vRO Client, open the Inventory browser, navigate to the designated lab ESXi host.
2. Drag the host into a temporary workflow input (`hostSystem` of type `VC:HostSystem`).
3. Create workflow `tmp-P0.4-Test5-ServiceTimestamp` with a Scriptable Task:

   ```javascript
   // P0.4 Test 5 — ESXi service start timestamp probe
   var serviceSystem = hostSystem.configManager.serviceSystem;
   var services = serviceSystem.serviceInfo.service;

   System.log("Total services: " + services.length);

   for (var i = 0; i < services.length; i++) {
       var svc = services[i];
       if (svc.key === "TSM-SSH") {
           System.log("=== TSM-SSH service ===");
           System.log("key: " + svc.key);
           System.log("label: " + svc.label);
           System.log("running: " + svc.running);
           System.log("policy: " + svc.policy);

           // Enumerate every property on the service object
           for (var prop in svc) {
               try {
                   var val = svc[prop];
                   if (typeof val !== "function" && typeof val !== "object") {
                       System.log("  " + prop + " = " + val);
                   }
               } catch (e) {
                   System.log("  " + prop + " — not readable");
               }
           }
       }
   }
   ```

4. Run the workflow. Capture log.
5. Inspect log for any timestamp-bearing field (e.g., `startedAt`, `lastChange`, `runtimeInfo.startTime`).
6. If no timestamp field is present on the service object, document this as a finding — Cleanup Workflow correlation will need to rely on alternative sources (e.g., ESXi `vmkernel.log` parsing via SSH, or `esxcli system syslog` queries).

### Expected Result (Two Possible Outcomes)

- **Outcome A (preferred)**: A timestamp field is present on the service object → Cleanup Workflow can use it directly.
- **Outcome B**: No timestamp field exposed → Cleanup Workflow falls back to log-based correlation.

Either outcome is informational and feeds into P8 (Cleanup Workflow design).

### Pass/Fail Criteria

- **PASS**: Workflow completes; service properties enumerated; operator documents which outcome applies.
- **FAIL**: Workflow exception accessing `serviceSystem.serviceInfo`.

### On Failure (Non-blocking)

Does not block subsequent tests. Records a design impact: investigate alternative property paths (`runtimeInfo`, `hostConfigInfo.service`) in P8.

### Rollback

None required. Delete `tmp-P0.4-Test5-ServiceTimestamp` workflow after sign-off.

### Evidence

- vRO log output showing full service property enumeration
- Operator's documented outcome (A or B) and field name if A

## Test 6 — SSH Plugin Smoke Test

**Mutation**: **Yes** — temporarily enables SSH on lab ESXi host, creates and removes a test local account, executes a benign `esxcli` command.
**Validates**: vRO SSH plugin can authenticate to ESXi via local account and execute commands, returning structured output.

### Prerequisites

- Test 2 PASSED (TCP/22 reachability)
- Designated lab ESXi host FQDN known
- vCenter endpoint registered in vRO inventory
- Operator has root credentials for lab ESXi host (used only for the prerequisite SSH enable; ephemeral account creation in production will use vSphere API per design)

### Procedure

#### Phase A — Manual SSH enable (preparatory; production uses API)

1. SSH to vCenter or use vSphere Client to enable SSH on the designated lab ESXi host:
   - vSphere Client → Host → Configure → Services → SSH → Start.
2. Confirm SSH service status `Running`.

#### Phase B — Create ephemeral test account via vCenter plugin

3. Create temporary workflow `tmp-P0.4-Test6-SSHSmoke` with input `hostSystem` of type `VC:HostSystem`.
4. Scriptable Task — create local account:

   ```javascript
   // P0.4 Test 6 Phase B — create ephemeral test account
   var accountManager = hostSystem.configManager.accountManager;
   var spec = new VcHostAccountSpec();
   spec.id = "vrorem-p04test";
   spec.password = "<ephemeral-test-password>"; // generate strong, single-use
   spec.description = "P0.4 Test 6 ephemeral";

   accountManager.createUser(spec);
   System.log("Account created: " + spec.id);

   // Grant Administrator role at root folder
   var authManager = hostSystem.configManager.authorizationManager;
   var rootFolder = hostSystem.parent; // climb to root — adjust per inventory

   // Output for next phase
   System.getContext().setAttribute("testAccount", spec.id);
   System.getContext().setAttribute("testPassword", spec.password);
   ```

5. Run Phase B. Confirm account creation in vSphere Client (Host → Configure → System → Local Users).

#### Phase C — SSH plugin command execution

6. Add a second Scriptable Task to the workflow:

   ```javascript
   // P0.4 Test 6 Phase C — SSH plugin command execution
   var sshSession = new SSHSession("<esxi-fqdn>", "vrorem-p04test");
   try {
       sshSession.connectWithPassword("<ephemeral-test-password>");
       sshSession.executeCommand("esxcli system version get", true);
       var output = sshSession.getOutput();
       var exitCode = sshSession.exitCode;
       System.log("Exit code: " + exitCode);
       System.log("Output:\n" + output);

       if (exitCode !== 0) {
           throw new Error("Non-zero exit code: " + exitCode);
       }
       if (output.indexOf("Version") < 0) {
           throw new Error("Output does not contain expected 'Version' field");
       }
   } finally {
       sshSession.disconnect();
   }
   ```

7. Run Phase C.

#### Phase D — Cleanup (rollback)

8. Add a third Scriptable Task:

   ```javascript
   // P0.4 Test 6 Phase D — remove ephemeral account
   var accountManager = hostSystem.configManager.accountManager;
   accountManager.removeUser("vrorem-p04test");
   System.log("Account removed: vrorem-p04test");
   ```

9. Run Phase D.
10. Manually disable SSH service via vSphere Client: Host → Configure → Services → SSH → Stop.
11. Confirm SSH service status `Stopped`.
12. Confirm `vrorem-p04test` is absent from Local Users list.

### Expected Result

- Phase B completes; account visible in vSphere Client.
- Phase C: SSH connection authenticates; `esxcli system version get` exits with code 0; output contains a `Version` field with the expected ESXi build.
- Phase D: account is removed; vSphere Client confirms absence.

### Pass/Fail Criteria

- **PASS**: All four phases complete as expected; ESXi version in output matches Pre-Execution Capture.
- **FAIL**: Any phase throws unhandled exception, SSH authentication fails, exit code non-zero, or output lacks expected fields.

## Test 7 — Plugin Method-Call Validation (8 Methods)

**Mutation**: **Yes** — exercises maintenance mode entry, SSH service start/stop, host configuration access on the designated lab ESXi host. Maintenance mode state is **deliberately preserved** at end of test for Test 8.
**Validates**: Each of the 8 plugin methods identified in P0.3 (replacing removed wrapper actions) is callable, returns the expected object type, and behaves as documented.

### Method Inventory Under Test

| # | Method | Object | Mutation |
|---|---|---|---|
| 7.1 | `host.config.product.fullName` | `VC:HostSystem` (property) | No |
| 7.2 | `clusterComputeResource.host` | `VC:ClusterComputeResource` (property) | No |
| 7.3 | `host.configManager.serviceSystem.startService("TSM-SSH")` | service | Yes (start) |
| 7.4 | `host.configManager.serviceSystem.stopService("TSM-SSH")` | service | Yes (stop) |
| 7.5 | `host.enterMaintenanceMode_Task(timeout, evacuate, spec)` | task | Yes (enter MM) — **preserved for Test 8** |
| 7.6 | `host.configManager.accountManager.removeUser(userName)` | account | Yes (test account already validated in Test 6 — re-validate signature only) |
| 7.7 | `host.exitMaintenanceMode_Task(timeout)` | task | **Deferred to combined rollback after Test 8** |
| 7.8 | `host.rebootHost_Task(force)` | task | **NOT executed in lab smoke test** — see note |

### Important Notes

- **Method 7.8 (`rebootHost_Task`) is intentionally not executed** in the smoke test. Rebooting a lab host disrupts other tests in flight and provides minimal incremental validation — the method signature is well-documented in vSphere API and the call shape is identical to other `_Task` methods. Validation is by inspection of API Explorer signature, not by execution.
- **Method 7.7 (`exitMaintenanceMode_Task`) is deferred** to the combined rollback after Test 8, because Test 8 needs the host in maintenance mode.
- **Method 7.5 (`enterMaintenanceMode_Task`)** is executed once and the state is shared with Test 8.

### Prerequisites

- Tests 1, 2, 6 PASSED
- Designated lab ESXi host FQDN known and host is **not** currently in maintenance mode
- Designated lab cluster A name known
- vCenter endpoint registered in vRO inventory
- All VMs on the designated host can be evacuated (vSAN cluster supports vMotion to peer hosts)

### Procedure

1. Create workflow `tmp-P0.4-Test7-MethodValidation` with inputs:
   - `hostSystem` (type `VC:HostSystem`)
   - `cluster` (type `VC:ClusterComputeResource`)

2. Single Scriptable Task — execute methods sequentially with logging between each:

   ```javascript
   // P0.4 Test 7 — plugin method-call validation
   System.log("=== 7.1 host.config.product.fullName ===");
   var fullName = hostSystem.config.product.fullName;
   System.log("fullName: " + fullName);
   if (!fullName || fullName.indexOf("ESXi") < 0) {
       throw new Error("7.1 FAIL — fullName missing or unexpected: " + fullName);
   }

   System.log("=== 7.2 clusterComputeResource.host ===");
   var clusterHosts = cluster.host;
   System.log("Cluster host count: " + clusterHosts.length);
   if (!clusterHosts || clusterHosts.length === 0) {
       throw new Error("7.2 FAIL — cluster.host empty or null");
   }
   for (var i = 0; i < clusterHosts.length; i++) {
       System.log("  Host: " + clusterHosts[i].name);
   }

   System.log("=== 7.3 startService TSM-SSH ===");
   var serviceSystem = hostSystem.configManager.serviceSystem;
   serviceSystem.startService("TSM-SSH");
   System.log("startService returned");
   // Verify
   var sshRunning = false;
   var services = serviceSystem.serviceInfo.service;
   for (var j = 0; j < services.length; j++) {
       if (services[j].key === "TSM-SSH") { sshRunning = services[j].running; break; }
   }
   System.log("TSM-SSH running: " + sshRunning);
   if (!sshRunning) { throw new Error("7.3 FAIL — TSM-SSH not running after startService"); }

   System.log("=== 7.4 stopService TSM-SSH ===");
   serviceSystem.stopService("TSM-SSH");
   System.log("stopService returned");
   // Verify
   var sshStopped = false;
   services = serviceSystem.serviceInfo.service;
   for (var k = 0; k < services.length; k++) {
       if (services[k].key === "TSM-SSH") { sshStopped = !services[k].running; break; }
   }
   System.log("TSM-SSH stopped: " + sshStopped);
   if (!sshStopped) { throw new Error("7.4 FAIL — TSM-SSH still running after stopService"); }

   System.log("=== 7.5 enterMaintenanceMode_Task ===");
   var mmSpec = new VcHostMaintenanceSpec();
   mmSpec.vsanMode = new VcVsanHostDecommissionMode();
   mmSpec.vsanMode.objectAction = VcVsanHostDecommissionModeObjectAction.ensureObjectAccessibility;
   var mmTask = hostSystem.enterMaintenanceMode_Task(900, true, mmSpec);
   System.log("enterMaintenanceMode_Task returned task: " + mmTask.id);

   // Wait for task completion using whichever method Test 4 identified
   // For this smoke test, poll task.info.state explicitly (works regardless of helper presence)
   var pollCount = 0;
   var maxPolls = 180; // 15 minutes at 5s intervals
   while (mmTask.info.state.value !== "success" && mmTask.info.state.value !== "error") {
       if (pollCount >= maxPolls) {
           throw new Error("7.5 FAIL — task did not complete within timeout");
       }
       System.sleep(5000);
       pollCount++;
   }
   if (mmTask.info.state.value === "error") {
       throw new Error("7.5 FAIL — task error: " + mmTask.info.error);
   }
   System.log("Maintenance mode entered. Task state: " + mmTask.info.state.value);

   // Capture maintenance mode entry timestamp candidates for Test 8
   var taskCompleteTime = mmTask.info.completeTime;
   System.log("Task completeTime: " + taskCompleteTime);
   System.getContext().setAttribute("test8_taskCompleteTime", taskCompleteTime);
   System.getContext().setAttribute("test8_taskId", mmTask.id);

   System.log("=== 7.6 accountManager.removeUser signature check ===");
   // Already exercised in Test 6 Phase D. Here we verify the method is exposed
   // and callable signature without invoking it (no test account currently present).
   var accountManager = hostSystem.configManager.accountManager;
   ```

## Test 8 — Maintenance Mode Entry Timestamp Retrieval

**Mutation**: No new mutation (uses state established by Test 7); combined rollback follows.
**Validates**: Two candidate methods for retrieving the timestamp at which a host entered maintenance mode. This is required by the Cleanup Workflow's correlation logic — distinguishing remediation-induced maintenance mode from administrator-induced.

### Candidate Methods Under Test

| # | Method | Source |
|---|---|---|
| A | `host.runtime.inMaintenanceMode` + `host.summary.runtime` properties | vSphere API direct |
| B | vCenter task history query for `EnterMaintenanceMode_Task` events on the host | Task collector / event manager |

### Prerequisites

- Test 7 PASSED (host is in maintenance mode)
- Task ID and completion time from Test 7 captured (`test8_taskCompleteTime`, `test8_taskId`)

### Procedure

   - `hostSystem` (type `VC:HostSystem`)

2. Scriptable Task:

   ```javascript
   // P0.4 Test 8 — maintenance mode timestamp retrieval

   System.log("=== Method A — host.runtime properties ===");
   System.log("inMaintenanceMode: " + hostSystem.runtime.inMaintenanceMode);

   // Enumerate runtime properties looking for any timestamp
   var runtime = hostSystem.runtime;
   for (var prop in runtime) {
       try {
           var val = runtime[prop];
           if (typeof val !== "function" && typeof val !== "object") {
               System.log("  runtime." + prop + " = " + val);
           } else if (val instanceof Date) {
               System.log("  runtime." + prop + " (Date) = " + val);
           }
       } catch (e) {
           System.log("  runtime." + prop + " — not readable");
       }
   }

   System.log("=== Method B — task history query ===");
   var taskManager = hostSystem.sdkConnection.taskManager; // adjust path per plugin
   var filter = new VcTaskFilterSpec();
   var entityFilter = new VcTaskFilterSpecByEntity();
   entityFilter.entity = hostSystem;
   entityFilter.recursion = VcTaskFilterSpecRecursionOption.self;
   filter.entity = entityFilter;

   var collector = taskManager.createCollectorForTasks(filter);
   try {
       var tasks = collector.readNextTasks(100);
       System.log("Recent tasks count: " + (tasks ? tasks.length : 0));

       var found = false;
       if (tasks) {
           for (var i = 0; i < tasks.length; i++) {
               var t = tasks[i];
               if (t.name && t.name.indexOf("EnterMaintenanceMode") >= 0) {
                   System.log("Found EnterMaintenanceMode task:");
                   System.log("  key: " + t.key);
                   System.log("  state: " + t.state.value);

   }
   ```

## Combined Rollback After Test 8

   ```javascript
   // Combined rollback — exit maintenance mode
   System.log("=== Exit maintenance mode ===");
   if (!hostSystem.runtime.inMaintenanceMode) {
       System.log("Host is not in maintenance mode; skipping exit.");
   } else {
       var exitTask = hostSystem.exitMaintenanceMode_Task(900);
       System.log("exitMaintenanceMode_Task returned: " + exitTask.id);

       var pollCount = 0;
       var maxPolls = 180;
       while (exitTask.info.state.value !== "success" && exitTask.info.state.value !== "error") {
           if (pollCount >= maxPolls) {
               throw new Error("Exit maintenance mode timeout");
           }
           System.sleep(5000);
           pollCount++;
       }
       if (exitTask.info.state.value === "error") {
           throw new Error("Exit maintenance mode failed: " + exitTask.info.error);
       }
       System.log("Maintenance mode exited. Task state: " + exitTask.info.state.value);
   }

   // Verify SSH service is stopped (Test 7 stopped it; confirm state)
   var serviceSystem = hostSystem.configManager.serviceSystem;
   var services = serviceSystem.serviceInfo.service;
   for (var i = 0; i < services.length; i++) {
       if (services[i].key === "TSM-SSH") {
           System.log("Final TSM-SSH running state: " + services[i].running);
           if (services[i].running) {
               System.warn("SSH still running — manually stop via vSphere Client");
           }
           break;
       }
   }

   // Verify no ephemeral test account remains
   System.log("Verify no vrorem-* accounts remain (manual check via vSphere Client)");
   ```
